## 后续 UI 动线设计原则

- FileChif 的工作台不应只围绕“选择 MD 文件转 Word”单一路径设计。
- 后续主工作台应围绕“输入来源 → 智能判定 → 输出目标 → 模板/字段确认 → 执行结果”的统一动线设计。
- 输入来源至少预留三类入口：
  - 拖拽文件：支持 `.md`、`.markdown`、`.txt`、`.docx` 等文件进入工作台。
  - 文件选择：保留当前本地文件选择能力。
  - 粘贴文本：支持从 AI 对话工具复制 Markdown 后直接粘贴到文本窗口。
- 拖拽交互建议：
  - 拖入 Markdown/TXT：默认建议输出 DOCX/PDF，并弹出模板选择。
  - 拖入 DOCX：后续可进入模板提取、样式检查、或 DOCX 转 PDF 等扩展任务。
  - 拖入未知文件：进入预检状态，提示可支持格式和下一步。
- 模板交互建议：
  - 不应长期隐藏在表单里；应在用户选择输出 DOCX 或识别到需要套版时主动弹出/侧滑选择。
  - 模板选择应支持“无模板”“最近模板”“默认模板”“模板库”。
- 粘贴交互建议：
  - 工作台提供“粘贴 Markdown”入口。
  - 粘贴后自动生成临时任务，允许用户选择保存源文本、输出格式、模板和目标路径。
- 为 AI 功能预留空间：
  - 后续 DeepSeek API 自动匹配字段应作为“模板字段确认”阶段的增强能力，而不是破坏主转换流程。
  - UI 上预留“字段识别/智能填充”区域，但默认不打扰纯转换用户。
